<objective>
Перевести `obligation_crawler_bonds` с append-only run-snapshot модели на модель
текущего состояния: одна строка на ticker через UNIQUE(ticker), плюс отдельная
лёгкая таблица истории цен `obligation_crawler_price_history`. Миграция должна
аккуратно схлопнуть уже существующие дубликаты ПЕРЕД добавлением UNIQUE-констрейнта
и сохранить read-контракт downstream API без изменений в коде API.

Purpose: Повторные прогоны краулера сейчас дублируют весь набор облигаций (плоский
INSERT без ON CONFLICT). Схема должна поддержать идемпотентный upsert по тикеру.
Output: Файл миграции migrations/002_*.sql, применяемый вручную через
`psql $DATABASE_URL -f` (механизм как у 001 — sqlx-миграций в проекте нет).

КРИТИЧНО — read-контракт API нельзя ломать. obligation-api (Diesel) читает данные
через вью `obligation_crawler_latest_bonds` и модель `LatestBond`. Вью ОБЯЗАН
продолжать возвращать тот же набор колонок в совместимых типах.

КРИТИЧНО — видимость mid-run. В старой append-only модели каждый прогон создавал
НОВЫЕ строки, а вью отбирал последнюю строку из уже ЗАВЕРШЁННОГО (`status='completed'`)
прогона — поэтому пока новый прогон шёл, предыдущая хорошая строка оставалась видна
API. В новой upsert-in-place модели (план 03) `save_bond` обновляет ту же строку и
перепривязывает её `run_id` к ТЕКУЩЕМУ прогону, который во время краулинга имеет
`status='running'` (см. obligation-crawler/src/services/bonds_crawler.rs: save_bond
вызывается в collect_bonds ~577 пока прогон 'running'; finish_crawl_run флипает в
'completed' только ~678-688 ПОСЛЕ цикла). Если оставить во вью фильтр
`WHERE r.status='completed'` (001_create_crawler_schema.sql:144), то КАЖДАЯ
обновлённая бумага исчезнет из API на всё время прогона и вернётся лишь по его
завершении — это реальная регрессия доступности read-контракта (CONTEXT помечает
read-контракт как КРИТИЧНО).

РЕШЕНИЕ (выбрано планировщиком): СНЯТЬ фильтр `status='completed'` (и сам JOIN на
obligation_crawler_runs) из переопределённого вью. Обоснование: после перехода на
current-state каждая строка bonds по определению И ЕСТЬ последний хороший скрейп
по тикеру — гейт `status='completed'` больше ничего не защищает, он лишь прячет
текущие данные на время прогона. Проверено по obligation-api/src/models/bonds.rs:
API (load_bond_page / load_bond_detail / count_bonds) делает только
`SELECT <17 колонок> FROM obligation_crawler_latest_bonds [WHERE/ORDER/LIMIT]` —
он НИКОГДА не ссылается на runs, status или семантику гейта, кроме «показать
последние хорошие данные». Колонки `run_id` и `created_at` сохраняются (API их
читает), но берутся прямо из строки bonds. Это самый простой вариант, который
держит контракт (альтернативы — не перепривязывать run_id до завершения прогона,
либо вести отдельный last_completed_run_id — сложнее и не дают преимуществ, раз
гейт ничего не защищает).
